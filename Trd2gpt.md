# Technical Requirements Document (TRD)

## Architecture Overview

Tracer consists of three primary components:

1. CLI tooling
2. CI execution agent
3. Static reporting dashboard

Tracer does not require a hosted backend service for v0.1.

---

## Core Data Artifacts

Tracer generates several structured artifacts during operation:

- `spans.jsonl`
- `change-summary.json`
- `doc-status.json`
- `doc-update.json`

These artifacts form the basis for reporting and auditability.

---

## Span Schema

Tracer records events using a JSONL span format.

Spans represent **observable AI tool interactions** rather than internal
model reasoning.

Example span structure:


{
"id": "span_123",
"timestamp": "...",
"tool": "cursor",
"eventType": "ai_call",
"model": "claude-sonnet",
"inputTokens": 800,
"outputTokens": 200,
"estimatedCost": 0.012,
"latencyMs": 450,
"metadata": {}
}


Tracer spans may optionally include:

- `traceId`
- `parentId`
- `commitHash`

These fields enable partial correlation across events.

Tracer does **not rely exclusively on commit hashes for correlation**.

---

## Correlation Strategy

CI-generated artifacts include explicit commit hashes.

Local spans may not always contain commit hashes.

Tracer correlates spans with commits using:

1. explicit commit metadata where available
2. session identifiers
3. approximate time windows

This approach provides **best-effort correlation**, not guaranteed
perfect attribution.

---

## Mapping Resolution

Mapping entries define relationships between code paths and documentation.

Example entry:


{
"code": "src/payments/**",
"doc": "docs/payments.md"
}


When multiple mappings match a file:

1. exact path matches win
2. narrower glob patterns win
3. later entries override earlier entries

If ambiguity remains, multiple documentation files may be considered.

---

## Documentation Update Generation

Tracer constructs prompts containing:

- code diff context
- related documentation
- mapping metadata

The AI model generates a proposed documentation update.

Tracer stores:

- full suggested content
- unified diff
- section metadata
- provenance information

Tracer attempts to avoid unsupported claims in generated documentation
but cannot guarantee perfect factual correctness.

Human review remains required.

---

## Delivery Strategies

Tracer supports multiple delivery modes:

### suggest

Produces a structured suggestion artifact.

Acceptance recorded via explicit user action.

---

### pr-comment

Posts suggestion in pull request comment.

Acceptance inferred when equivalent change appears in merged commit.

---

### commit

Commits documentation change to branch.

Acceptance recorded when merged into default branch.

---

## Idempotency

Tracer attempts to avoid duplicate suggestions.

Suggested content is hashed to detect identical updates.

Because AI models are probabilistic, identical prompts may still produce
different outputs.

Tracer therefore treats idempotency as **best effort**, not guaranteed.

---

## Documentation Status

`doc-status.json` records documentation state.

State is defined relative to the **default branch**.

Branch-specific suggestions may temporarily diverge until merged.

States include:

- CURRENT
- PENDING
- OUTDATED

---

## Static Dashboard

The dashboard is implemented as a static site.

Data sources include:

- span reports
- change summaries
- documentation status artifacts

The dashboard reads JSON artifacts generated by CI.

No persistent backend service is required for v0.1.

---

## Storage

v0.1 storage uses simple artifact files.

Examples:


.tracer/spans.jsonl
.tracer/doc-status.json
.tracer/change-summaries/


Future versions may introduce optional persistent storage.

---

## Security

Tracer stores only minimal metadata about AI tool usage.

Sensitive prompt content is not stored in span logs.

Teams may disable span collection entirely if required.

---

## Observability Integration

Tracer may export span data to external systems in the future.

Potential targets include:

- OpenTelemetry pipelines
- AI observability platforms

v0.1 focuses on local artifact generation.

---

## Failure Handling

Tracer handles failures conservatively:

- AI generation failures produce no doc updates
- span collection failures do not block development workflows
- malformed artifacts are ignored

The system prioritizes **developer workflow safety** over completeness of
reporting.
